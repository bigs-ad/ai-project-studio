# Game Profile

Optimize for proving player experience before producing systems or content.

## Phase focus

- discovery: define target player, game promise, core loop, session shape, constraints, and explicit non-goals.
- validation: build the smallest playable loop and observe players. Test whether choices, feedback, challenge, and reward are understandable and engaging.
- build: create one representative vertical slice before multiplying levels, characters, enemies, items, or narrative content.
- release: verify distributable builds, input and device support, saves when applicable, performance budgets, onboarding, and crash recovery.
- operate: separate observed player behavior from interpretation, then choose the next hypothesis.

## Ordering rules

Do not prioritize progression, equipment, pets, large content sets, polish, or live systems until the core loop has credible playtest evidence. Do not treat the creator's own repeated play as sufficient external validation. Prefer a crude playable artifact over a detailed design document when testing fun or clarity.

## Review questions

- Can a new player understand the immediate goal without explanation?
- Does the main action create clear audiovisual and mechanical feedback?
- Are decisions meaningful rather than merely available?
- Does failure teach something and invite another attempt?
- Is content production affordable for one developer?
- Does the build stay within the declared performance and platform budget?

## In-engine quality evidence

For a game-facing visual or interaction deliverable, distinguish concept/reference images from in-engine results. Before calling a vertical slice, production asset, or final-in-context deliverable accepted, inspect the applicable evidence:

- input states and readable mechanical feedback;
- motion timing, easing, transitions, and camera response;
- lighting, shadow, depth, material, and compositing behavior;
- VFX and audio response where they carry game feel;
- target resolution, device, and performance behavior;
- a screenshot or recording from the running build, not only a source image.

Apply only the checks relevant to the approved contract. Placeholder work may omit polish, but the omission and remaining steps must be explicit.
